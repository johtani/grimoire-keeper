# Grimoire Keeper 改善点一覧

**作成日**: 2024年12月  
**分析者**: Amazon Q

## 🔴 緊急度：高 (Critical)

### 1. セキュリティ脆弱性

#### 1.1 CORS設定が緩すぎる
- **現状**: `allow_origins=["*"]` で全てのオリジンを許可
- **リスク**: CSRF攻撃、データ漏洩
- **対応**: 
  - 本番環境では特定のドメインのみ許可
  - 環境変数で管理 (`ALLOWED_ORIGINS`)
- **ファイル**: `apps/api/src/grimoire_api/main.py`

#### 1.2 認証・認可機能が未実装
- **現状**: 誰でもAPIにアクセス可能
- **リスク**: 不正アクセス、データ改ざん
- **対応**:
  - API Key認証の実装
  - JWT認証の検討
  - エンドポイント毎のアクセス制御
- **優先度**: 最高

#### 1.3 レート制限なし
- **現状**: リクエスト数の制限なし
- **リスク**: DoS攻撃、リソース枯渇
- **対応**:
  - `slowapi`または`fastapi-limiter`の導入
  - IP/ユーザー単位での制限
- **推奨**: 100 req/min/IP

#### 1.4 入力値検証が不十分
- **現状**: Pydanticの基本検証のみ
- **リスク**: SQLインジェクション、XSS
- **対応**:
  - URL検証の強化
  - サニタイゼーション処理
  - 最大長制限の明示

### 2. データベース・キャッシュファイルの混入

#### 2.1 データベースファイルがコミット済み
- **現状**: `grimoire.db`がリポジトリに存在
- **問題**: 個人データの漏洩リスク
- **対応**:
  ```bash
  git rm --cached grimoire.db apps/api/grimoire.db
  git commit -m "Remove database files from repository"
  ```

#### 2.2 データファイルの混入
- **現状**: `data/json/*.json` (18ファイル)がコミット済み
- **問題**: 個人の閲覧履歴が公開される
- **対応**:
  ```bash
  git rm --cached data/json/*.json
  git commit -m "Remove personal data files"
  ```

#### 2.3 キャッシュファイルの混入
- **現状**: `.mypy_cache/`, `.ruff_cache/`がコミット済み
- **問題**: リポジトリサイズの肥大化
- **対応**:
  ```bash
  git rm -r --cached .mypy_cache .ruff_cache
  git commit -m "Remove cache directories"
  ```

### 3. 環境変数の必須チェック不足
- **現状**: 起動時に環境変数の検証なし
- **リスク**: 実行時エラー、予期しない動作
- **対応**:
  - 起動時バリデーション関数の実装
  - 必須変数リストの明示
  - エラーメッセージの改善

## 🟡 緊急度：中 (Medium)

### 4. 共有ライブラリの未実装

#### 4.1 共通モデルの不在
- **現状**: `shared/`にtelemetryのみ
- **問題**: API/Bot間でコード重複
- **対応**:
  - `PageModel`, `SearchResult`等の共通化
  - `shared/src/grimoire_shared/models/`の作成

#### 4.2 ユーティリティ関数の重複
- **対応**:
  - 日付処理、バリデーション等を共通化
  - `shared/src/grimoire_shared/utils/`の作成

#### 4.3 設定管理の不統一
- **対応**:
  - 共通設定クラスの作成
  - 環境変数読み込みの統一

### 5. エラーハンドリングの改善

#### 5.1 カスタム例外の拡充
- **現状**: 基本的な例外のみ
- **対応**:
  - `DatabaseError`, `ExternalAPIError`等の追加
  - エラーコードの体系化
- **ファイル**: `apps/api/src/grimoire_api/utils/exceptions.py`

#### 5.2 ログ機能の標準化
- **現状**: print文とloggingが混在
- **対応**:
  - 構造化ログの導入 (`structlog`)
  - ログレベルの統一
  - ログローテーション設定

#### 5.3 エラーレスポンスの統一
- **対応**:
  - 統一エラーレスポンスモデル
  - HTTPステータスコードの適切な使用
  - エラー詳細の国際化対応

### 6. テスト環境の改善

#### 6.1 統合テスト設定の複雑さ
- **現状**: `conftest_integration.py`が別ファイル
- **対応**:
  - `conftest.py`への統合
  - フィクスチャのスコープ最適化

#### 6.2 モックの重複
- **対応**:
  - 共通モックフィクスチャの作成
  - `pytest-mock`の活用強化

#### 6.3 テストデータ管理
- **対応**:
  - テストデータファクトリの実装
  - `factory_boy`の導入検討

### 7. パフォーマンス最適化

#### 7.1 データベースインデックス未設定
- **現状**: 主キー以外にインデックスなし
- **影響**: 検索クエリが遅い
- **対応**:
  ```sql
  CREATE INDEX idx_pages_url ON pages(url);
  CREATE INDEX idx_pages_status ON pages(status);
  CREATE INDEX idx_pages_created_at ON pages(created_at);
  CREATE INDEX idx_logs_page_id ON processing_logs(page_id);
  ```

#### 7.2 キャッシュ機能なし
- **対応**:
  - Redis導入の検討
  - インメモリキャッシュ (`cachetools`)
  - 検索結果のキャッシュ (TTL: 5分)

#### 7.3 バッチ処理の最適化
- **対応**:
  - 並列処理の導入
  - バルクインサートの活用
  - トランザクション最適化

### 8. OpenTelemetry依存の見直し
- **現状**: 全サービスで必須
- **問題**: 開発環境で不要な場合も
- **対応**:
  - オプショナル化
  - 環境変数での有効/無効切り替え
  - デフォルトは無効
