GitHub Issue を分析して実装計画を立て、作業準備を整えます。

## 手順

1. **Issue を読み込む**
   - 引数が URL の場合: `gh issue view <URL>`
   - 引数が番号の場合: `gh issue view <番号>`
   - Issue のタイトル・本文・ラベル・コメントを把握する

2. **コードベースを調査する**
   - Issue の内容に関連するファイルを特定する
   - 影響範囲を把握する (サービス / リポジトリ / モデル / ルーター)
   - 既存のテストファイルを確認する

3. **実装計画を提示する**
   以下の形式で計画をまとめてユーザーに承認を求める:
   ```
   ## 実装計画: <Issue タイトル>

   ### 変更対象ファイル
   - `path/to/file.py` — 変更内容の概要

   ### 実装ステップ
   1. ...
   2. ...

   ### テスト方針
   - 追加・修正するテストの概要

   ### 懸念点 (あれば)
   - ...
   ```

4. **ユーザーの承認を得てからブランチを作成する**
   - ブランチ名: `feat/issue-<番号>-<短い説明>` or `fix/issue-<番号>-<短い説明>`
   - `git checkout main && git pull origin main && git checkout -b <ブランチ名>`

## 注意事項
- 計画を提示した後、必ずユーザーの確認を待ってからブランチを作成すること
- 実装は行わない — 計画と準備のみ
- スコープを Issue の内容に厳密に限定する
